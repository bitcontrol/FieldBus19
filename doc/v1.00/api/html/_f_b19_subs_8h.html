Detailed Description

This file publishes the interface of the FieldBus19 Subscriber library.

  • Filename: FB19Subs.h
  • Platform(s): All
  • Project: FB19SubsLib
  • Created: May 23, 2023

  • Description: This file publishes the interface of the FieldBus19 Subscriber library.

    A note on the parameter instance: The parameter instance can be thought of as a 'handle' to the Subscriber instance. It is the start address of the memory area, where the instance stores its internal values (buffers, counters, pointers, ...).

    It doesn't matter for this library if the memory has been allocated statically, dynamically or on the stack. Just make sure it's not deallocated before the FB19 Subscriber is stopped by a call to FB19Subs_stop(), or the behavior of the Subscriber will be undefined.
    That way, this library doesn't have to allocate memory internally; all required memory space is passed from the outside.

    Limitation of the bit rate:
    This version of the FB19 Subscriber library is limited to a maximum bit rate of 250kbit/s, although many of STM's microcontroller families could run the bus at considerably higher rates.

    Limitation of the number of Subscribers:
    This version of the FB19 Subscriber library is limited to a maximum number of 8 Subscribers, although it could by design handle up to 30 Subscribers.

    Code size and stack requirements:
    This library was designed with the economic use of resources in mind. It occupies little FLASH memory and uses the stack economically. Please consult the output of your linker and debugger for detailed and accurate information.

◆ FB19Subs_handler()

void FB19Subs_handler ( FB19Subs_inst_t *  instance)

Executes the periodic tasks of this library.

It essentially moves unread messages that came in since the previous _handler() call aside, fetches the received messages from the driver layer and copies them to the next higher layer, and finally copies the messages to be transmitted to the driver layer.

Set the period between the calls equal to the period on the FB19 Controller. Since the _handler() calls on the FB19 Controller and its subscribers can't be synchronized, some will call their _handler() function a bit more frequent than others.
If the FB19 Controller's _handler() function is called more frequent than that of one of its subscribers, every now and then a response message will be returned unchanged, which indicates this scenario.
If the FB19 Controller's _handler() function is called less frequent than that of one of its subscribers, a subscriber will every now and then return R_ERROR when its _submit() function is called. This indicates that the transmit queue was already full before the call and indicates this scenario.

When it is known for sure that a FB19 Subscriber receives messages from its FB19 Controller only every Nth Controller's _handler() cycle, the period of the Subscriber's _handler() calls can be N times of that of the Controller's period.

Moving unread messages to a different buffer prevents 'message loss' when frequent message writer and reader objects (classes, modules, tasks) compete with infrequent writer and reader objects on the available space in the receive message queue. The frequent writers/readers would overwrite (delete) messages of the infrequent writers/readers, since the latter may pick up their response messages just every 2nd, 3rd or xth _handler() call.

It does nothing if instance is a NULL pointer or if the library hasn't been started yet.

Parameters
[in]instancepoints to the memory resource used by this instance.

◆ FB19Subs_start()

int FB19Subs_start ( FB19Subs_inst_t *  instance,
const FB19Subs_cfg_t *  config,
int  bitRate 
)

Starts an FB19 Subscriber instance.

The parameter instance can be thought of as a 'handle' to the Subscriber instance. It is the start address of the memory area, where the instance stores its internal values (buffers, counters, pointers, ...).

The parameter config contains system on chip configuration parameters as well as transmit and receive message buffers. It is left to the user to define how large the buffers shall be. Systems with high message rate peaks and low CPU power require larger buffers than those with steady data exchange.

Parameters
[in]instancepoints to the memory resource used by this instance.
[in]configpoints to the structure that configures this instance. It contains system on chip configuration parameters as well as pointers to the transmit and receive message buffers. It is left to the user to define how large the buffers shall be. For example, systems with high message rate peaks and low CPU power require larger buffers than those with steady data exchange.
[in]bitRateis the number of bits per second during character reception and transmission.
Returns
R_SUCCESS if the parameter checks pass, the module instance hasn't been started already and could be started with the passed configuration, R_ERROR otherwise.

◆ FB19Subs_submit()

int FB19Subs_submit ( FB19Subs_inst_t *  instance,
const FB19Msg_t *  msg 
)

Deep copies the message msg points to into the transmit message queue of this library. After the next call to the _handler() function, it is ready for transmission.

It will however only be sent when the FB19 Controller sends this node a request message that matches the message identifier msgId inside its descriptor dscr.

The messages are transmitted in the order they have been added through the _submit() calls.

When the transmit message buffer is full before this function is called, it returns R_ERROR. It also returns this value if the descriptor desc of the message msg points to contains implausible data. Its fields dstAddr, msgId and pldSize are checked for their range when this function is called.

Parameters
[in]instancepoints to the memory resource used by this instance.
[in]msgpoints to the message that will be added to the transmit queue if this function returns successfully.
Returns
R_SUCCESS if the parameter checks pass, the message msg points to contains plausible data in the descriptor, and the transmit queue had the capacity to store this message, R_ERROR otherwise.
